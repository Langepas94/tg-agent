# Демо для заказчика: RAG с источниками/цитатами/«не знаю» (задание 4) и диалог с памятью задачи (задание 5)

Живая демонстрация в Telegram-боте (прод, VPS) + командные доказательства.
Вся демка — 5–7 минут. Сообщения копируются как есть, по порядку.

База знаний бота: документация и код проектов `tg-agent` и `open-meteo-mcp`.

---

## Часть 1 — задание 4: источники, цитаты, «не знаю»

**Тезис:** каждый ответ обязан содержать список источников
(`файл / раздел #chunk_id`) и дословные «цитаты» из найденных чанков; если
релевантность ниже порога — ассистент обязан отказаться и попросить уточнение
(отказ зашит в код, LLM в этой ветке вообще не вызывается — «уговорить»
модель ответить невозможно).

| # | Отправить боту | Что показать заказчику |
|---|---|---|
| 1.1 | `/rag on` | Ответ `✅ RAG mode enabled…` — обычные вопросы пошли через базу знаний |
| 1.2 | `Как бот подключает MCP-серверы в рантайме?` | Ответ + блок **Источники:** — каждый пункт вида `[1] tg-agent/README.md / раздел #chunk_id score=0.87` и под ним дословная «цитата» из документа |
| 1.3 | `Какие переменные окружения нужны tg-agent для запуска?` | То же: ответ опирается на `.env.example`/README, в источниках видны реальные файлы и цитаты |
| 1.4 | `Кто выиграл чемпионат мира по футболу 2018 года?` | **«Не знаю. В базе знаний не нашлось достаточно релевантного материала… Уточните, пожалуйста, вопрос»** — и НИ ОДНОГО источника. Вопрос вне базы срезан порогом релевантности (0.5 по косинусной близости) |

Акцент для заказчика на шаге 1.4: это не вежливость модели, а фильтр в коде —
ниже порога запрос до LLM не доходит, галлюцинация невозможна по построению.

### Проверка «на 10 вопросах» (усиление задания 4)

Контрольный сет из 10 вопросов с ожиданиями и эталонными источниками прогнан
целиком; отчёт лежит в репо RAG-движка:

- `reports/rag_answer_comparison.md` — сводка: **источники в 10/10 ответов
  (hit@5 = 1.0), цитаты в 10/10 (quotes rate = 1.0)**, покрытие ожиданий
  RAG 0.63 против 0.38 без RAG;
- `reports/rag_answer_comparison.json` — по каждому вопросу: оба ответа
  (plain/RAG), источники с цитатами, метрики совпадения смысла.

Показать терминалом (по желанию):

```bash
# контракт ответа: relevant, sources[].quote — прямо с прод-сервера
ssh -i ~/.ssh/id_ed25519_vps root@5.129.234.9 \
  'set -a; source /opt/tg-agent/.env; set +a; \
   $RAG_INDEXER_BIN answer --mode rag --json --index $RAG_INDEX \
     --query "Как бот подключает MCP-серверы?" \
     --model $RAG_EMBED_MODEL --chat-model $RAG_CHAT_MODEL \
     --chat-url $RAG_CHAT_URL --chat-provider $RAG_CHAT_PROVIDER \
     --search-mode hybrid --top-k 5 --min-dense-score $RAG_MIN_SCORE' \
  | python3 -m json.tool | head -40
```

## Часть 2 — задание 5: диалог с историей, RAG на каждом ходе, память задачи

**Тезис:** бот хранит историю диалога, на каждый вопрос заново ищет контекст
в базе, отвечает с учётом найденного и всегда выводит источники; параллельно
ведёт «память задачи» — цель диалога, что уже уточнено, зафиксированные
ограничения/термины.

Продолжаем ту же сессию (после части 1 можно `/reset` и заново `/rag on`,
чтобы память была чистая).

| # | Отправить боту | Что показать заказчику |
|---|---|---|
| 2.1 | `Что такое tg-agent и из чего он состоит?` | Обычный ответ с источниками — старт диалога, цель зафиксировалась |
| 2.2 | `А какие транспорты он поддерживает?` | **«он»** разрешилось из истории (история диалога + query rewrite): ответ про HTTP Streamable и stdio у tg-agent, источники снова на месте |
| 2.3 | `Как агент хранит память о пользователе?` | Ответ про слои памяти (sticky-facts), источники |
| 2.4 | `Какие слои у этой памяти?` | **«этой»** — снова отсылка назад; ответ по слоям short-term/working/long-term с источниками |
| 2.5 | `/rag status` | Блок **Task state**: `Цель диалога: …` + `Уже уточнено: …` — бот показывает память задачи, которую вёл всё это время |
| 2.6 | `Что делает команда /watch?` | Диалог продолжается, цель не потеряна, источники в каждом ответе |
| 2.7 | `/reset` | `✅ Chat memory reset…` — история и task state очищены; следующий `/rag status` покажет пустую память |

Акценты для заказчика:

- шаги 2.2/2.4 — вопросы с местоимениями работают ТОЛЬКО если бот реально
  передаёт историю диалога в поиск (без неё «он»/«этой» не найдут ничего);
- шаг 2.5 — task state не пишется руками: его ведёт отдельный LLM-экстрактор
  перед каждым ответом;
- источники есть в каждом ответе на всём протяжении диалога.

### Проверка «на 2 длинных сценариях» (усиление задания 5)

Автотесты гоняют два полных диалога — 12 и 10 сообщений — через тот же
код, что обрабатывает сообщения в Telegram, против реального индекса. На
каждом ходе проверяется: релевантный ответ обязан нести источники и цитаты,
отказ обязан быть без источников, цель диалога дожила до конца:

```bash
cd tg-agent
RAG_INDEX=~/Documents/Rag/ollama-rag-indexer/indexes-real-qwen/structural \
RAG_INDEXER_BIN=~/Documents/Rag/ollama-rag-indexer/.venv/bin/rag-indexer \
cargo test --test live_rag_dialog -- --ignored --nocapture --test-threads=1
```

Результат последнего прогона: оба сценария зелёные — 11 и 9 ответов с
источниками+цитатами, по одному честному отказу на офф-топик вопрос,
цель диалога удержана (см. `tests/live_rag_dialog.rs`).

## Если что-то пошло не так во время демо

- Бот отвечает «RAG включен, но клиент не настроен / индекс не найден» —
  проверить `RAG_*` в `/opt/tg-agent/.env` и `systemctl restart tg-agent`.
- Первый вопрос после долгого простоя думает 30–60 сек — Ollama на сервере
  поднимает embedding-модель с диска (2GB боксу не хватает RAM держать её
  резидентно). Дальше ходы идут ~10 сек. Перед демо разогреть одним вопросом.
- `❌ RAG error: …` с таймаутом — повторить сообщение; DeepSeek API изредка
  тормозит.
